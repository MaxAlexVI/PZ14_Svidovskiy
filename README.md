# Практическое задание №14

## Тема

Реализация очереди задач по модели producer-consumer.

## Цель работы

Реализовать очередь фоновых задач через RabbitMQ, добавить повторные попытки обработки, ограничение числа попыток, dead-letter queue и идемпотентность worker.

## Задание по методичке

В рамках практической работы необходимо:

- поднять RabbitMQ;
- создать основную очередь задач и DLQ;
- реализовать endpoint для постановки job в очередь;
- реализовать worker по модели producer-consumer;
- добавить поле `attempt`;
- реализовать ограничение числа попыток;
- реализовать перевод сообщения в DLQ при превышении лимита попыток;
- реализовать идемпотентную проверку по `message_id`;
- показать успешную обработку и обработку с ошибкой.

## Сценарий работы

1. Клиент отправляет HTTP-запрос на постановку задачи в очередь.
2. Сервис `tasks` формирует сообщение и публикует его в очередь `task_jobs`.
3. Worker читает сообщение из очереди.
4. Worker проверяет, не было ли сообщение уже обработано ранее.
5. Worker выполняет имитацию тяжелой операции.
6. При успехе сообщение подтверждается через `ack`.
7. При ошибке сообщение отправляется на повторную попытку или в DLQ.
8. Если число попыток превышено, сообщение оказывается в `task_jobs_dlq`.

## Что реализовано

- Поднят RabbitMQ через `deploy/rabbit/docker-compose.yml`.
- Созданы очереди `task_jobs` и `task_jobs_dlq`.
- Реализован endpoint `POST /v1/jobs/process-task`.
- Сообщение задачи содержит `job`, `task_id`, `attempt`, `message_id`.
- Worker проверяет идемпотентность по `message_id`.
- При ошибке worker увеличивает `attempt` и публикует задачу повторно.
- После превышения лимита попыток задача отправляется в `task_jobs_dlq`.

## Как работает

Producer формирует `TaskJob`, ставит `attempt = 1`, генерирует `message_id` и публикует сообщение в `task_jobs`.

Worker читает очередь с ручным подтверждением. Если задача уже обработана, она подтверждается через `ack` без повторной работы. Если обработка завершилась ошибкой, worker либо отправляет задачу на повторную попытку, либо переносит ее в DLQ после превышения лимита попыток.

## Ожидаемый результат

Для `task_id = t_001` worker успешно обрабатывает задачу и подтверждает сообщение. Для `task_id = t_fail` worker имитирует ошибку, выполняет несколько попыток и после лимита публикует сообщение в `task_jobs_dlq`. В RabbitMQ Management UI можно увидеть основную очередь и DLQ.

## Ответы на контрольные вопросы

- Задача в очереди описывает работу, которую нужно выполнить, а событие сообщает о факте, который уже произошел.
- Retries нужны, чтобы переживать временные ошибки обработки.
- Нельзя бесконечно возвращать ошибочное сообщение в очередь, потому что оно будет блокировать обработку и создавать бесконечный цикл.
- DLQ используется для проблемных сообщений, которые не удалось обработать после заданного числа попыток.
- Повторная доставка возможна при сбоях worker или отсутствии `ack`.
- Идемпотентность означает, что повторная обработка одного сообщения не ломает состояние системы.
- `message_id` нужен для распознавания повторно доставленного сообщения.
- Хранение обработанных `message_id` даже в памяти показывает принцип защиты от дублей.
- Если worker выполнил работу, но не отправил `ack`, RabbitMQ может доставить сообщение повторно.
- Producer-consumer удобен для тяжелых фоновых задач, потому что HTTP-сервис быстро ставит работу в очередь, а worker выполняет ее отдельно.

## Вывод

В ходе работы была реализована надежная схема фоновой обработки задач. Retry помогает переживать временные ошибки, DLQ отделяет проблемные сообщения, а `message_id` демонстрирует принцип идемпотентной обработки.
